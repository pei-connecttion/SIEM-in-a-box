# SIEM-in-a-Box

**A Lightweight Architecture for Security Information and Event Management**

A modular, open-source SIEM architecture for **SMEs and educational environments**, built on the Elastic Stack and Apache Kafka. Designed for resource-constrained deployments with limited budgets and security staffing, it delivers enterprise-grade monitoring, detection, and alerting from a single `docker compose up` command.

---

## Architecture Overview

### Five-Layer Design

The architecture is organized into five distinct layers, each with a clear responsibility:


| Layer | Responsibility | Key Components |
|---|---|---|
| **Ingestion** | Collect telemetry from heterogeneous sources | Elastic Agent, Filebeat, Kafka |
| **Processing** | Parse, normalize (ECS), and enrich raw events | Logstash (Grok, date, useragent, mutate) |
| **Storage/Correlation** | Near real-time indexing, search, and event correlation | Elasticsearch |
| **Management/Control** | Policy distribution, agent lifecycle, enrollment | Fleet Server, Kibana Fleet UI |
| **Alerting/Response** | Scheduled detection rules, triage, case management | Kibana Detection Engine, Cases, Timelines |

### Dual Ingestion Strategy

The system supports two complementary ingestion modes to cover both managed endpoints and legacy/agentless sources:

| Mode | Transport | Buffer | Latency | Delivery Guarantee | Use Case |
|---|---|---|---|---|---|
| **Agent-Ingest** | Elastic Agent -> Elasticsearch (direct) | Local only (on-agent) | Low (near real-time) | Best-effort | Managed workstations, servers |
| **Push-Ingest** | Syslog -> Filebeat -> Kafka -> Logstash -> Elasticsearch | Kafka (durable, distributed) | Slightly higher | Guaranteed (Kafka retention) | Legacy systems, firewalls, network devices, IoT |

---

## Data Flow (Push-Ingest)

1. **Collection**: Event sources transmit logs via syslog (TCP/UDP) to Filebeat collectors.
2. **Buffering**: Kafka (KRaft mode) acts as a durable message buffer, decoupling producers from downstream processing. During Elasticsearch overloads, maintenance, or pipeline failures, Kafka retains messages until processing capacity is restored.
3. **Processing**: Logstash consumes from Kafka and applies an ETL pipeline -- Grok parsing, ECS normalization, timestamp standardization, user-agent extraction, and log-level classification. Parse failures are tagged (not dropped) to preserve non-conforming events for review.
4. **Storage**: Processed events are indexed in Elasticsearch with near real-time search capability (<1 second to searchable).
5. **Analysis**: Kibana provides dashboards, interactive investigation, rule-driven alerting, and case management.

## Ingestion Modes In Depth

### Agent-Ingest

Elastic Fleet agents are installed directly on monitored endpoints (workstations, servers). The Fleet Server provides:

1. **Centralized lifecycle management** -- creation, configuration distribution, health monitoring
2. **Dynamic policy distribution** -- real-time configuration changes for the entire agent fleet
3. **Secure communication** -- authenticated channel between agents and the Elasticsearch cluster

Agents store logs locally and transmit them in real time directly to Elasticsearch. Since there is no centralized buffer, Agent-Ingest **prioritizes low latency over guaranteed delivery** -- sustained outages exceeding local buffer capacity can drop events. This trade-off suits use cases where near-real-time visibility matters more than complete retention.

### Push-Ingest

Supports systems where agents are unavailable or impractical (legacy systems, network devices, third-party services, ephemeral containers). The pipeline:

1. Event sources transmit logs via **syslog** (TCP/UDP) to Filebeat on port 5140
2. Filebeat tags events with source metadata and publishes to a Kafka topic
3. **Kafka retains messages** regardless of downstream state -- during Elasticsearch overload, maintenance, or pipeline failures, events are preserved in Kafka's distributed log
4. Logstash consumes and applies ETL transformations
5. Processed events are indexed in Elasticsearch

This provides **guaranteed delivery**: events remain available for consumption regardless of transient failures.

## Configuration Management (Fleet)

The Management and Control Layer centralizes policy-driven configuration for distributed Elastic Agents, avoiding per-host manual configuration.

Agents enroll using **time-limited enrollment tokens** generated by Fleet Server. After enrollment, Fleet Server continuously propagates policy updates and tracks agent health, while event data flows directly to Elasticsearch.

### Pre-configured Agent Policies

| Policy | ID | Namespace | Purpose |
|---|---|---|---|
| **Fleet Server Policy** | `fleet-server-policy` | `default` | Manages the Fleet Server itself |
| **Linux System Monitoring** | `linux-system-policy` | `production` | Endpoint monitoring for Linux hosts |

**Linux System Monitoring** collects:

- **Metrics**: CPU, memory, network, filesystem, disk I/O
- **Logs**: Authentication logs (`/var/log/auth.log`, `/var/log/secure`), syslog (`/var/log/messages`, `/var/log/syslog`)

### Enrolling External Agents

```bash
sudo elastic-agent enroll \
  --url=https://<fleet-server-host>:8220 \
  --enrollment-token=<token-from-kibana> \
  --certificate-authorities=<path-to-ca.crt>
```

## Quick Start

```bash
# 1. Clone the repository
git clone https://github.com/ieeta-pt/SIEM-in-a-box.git
cd SIEM-in-a-box

# 2. Review and customize environment variables (passwords, ports, memory)
vim stack/.env

# 3. Deploy the stack
cd stack
docker compose up -d

# 4. Monitor startup
docker compose logs -f
```
